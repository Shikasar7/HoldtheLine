## 1. 总体架构：服务器权威 + 同一个宿主

联机的核心思想只有一句话：**客户端只说"我想"，服务器回"发生了什么"。**

```
┌─ 玩家电脑 ────────────────────────────────────────────┐
│ BattleScene (Godot UI)                                │
│   │  只面向 IGameHost 接口，不知道对面是本地还是网络      │
│ RemoteGameHost (IGameHost 的联机实现)                  │
│   │  缓存 PlayerView / 合法着法；提交命令、接收事件       │
│ GameServerClient — 协议层：JSON 消息、退避重连、resume  │
│   │                                                   │
│ WebSocketTransport (IClientTransport 的 WS 实现)       │
└───┼───────────────────────────────────────────────────┘
    │ wss://htl.cicala.chat/ws   (TLS 加密，走 443)
┌─ VPS ──────┼──────────────────────────────────────────┐
│ nginx :443 — TLS 终止，按域名分流，转发 /ws             │
│ Kestrel :5210 (ASP.NET，systemd 常驻)                  │
│   ├ 握手：协议版本 + DataHash + 账号验证                │
│   ├ QueueManager：ELO 就近配对（K=32，新号前 10 局 64）  │
│   └ MatchSession（每局一个）                            │
│       └ LocalGameHost ← 与单机人机用的是同一个类         │
│ SQLite /var/lib/holdtheline/holdtheline.db             │
│   (账号/卡组/收藏/天梯) + match-logs/ + 每日备份         │
└───────────────────────────────────────────────────────┘
```

四条原理：

1. **服务器权威（server-authoritative）**。客户端从头到尾没有完整 `GameState`，只有按座位裁剪
   的 `PlayerView`——对手手牌数据根本没发到你机器上，透视外挂在协议层面就不存在。客户端也不执行
   规则：它提交意图（`Command`），服务器上的规则引擎判定合法性并结算，把结果事件推回来。
   改客户端内存改不了战局。

2. **命令/事件流**。出一张牌 = 一条 `Command` JSON 上行 → `MatchSession` 经单消费者 Channel
   排队执行（天然防并发竞态）→ 结算出一批 `GameEvent` → 按两个座位各自裁剪后分别推送 →
   客户端把事件队列播成动画。事件批次附带最新 PlayerView 与该座位的合法着法列表，
   所以"哪些格能放"的高亮也是服务器告诉的。

3. **服务器上的对局宿主就是 `LocalGameHost`**——与人机对战完全同一个类。这是 P0 起
   "联机地基五硬约束"的回报（IGameHost 异步接口、命令/事件序列化往返、回环序列化、
   宿主层按座位裁剪、确定性 RNG+命令日志）：联机只是把宿主从本地进程搬到 VPS，
   中间插一条 WebSocket，规则代码零改动。Steam 将来也只作为 `IClientTransport`
   的平替传输层接入，不改会话逻辑（见 docs/05 尾注的 Steam 决定）。

4. **一切皆可重放**。确定性 RNG + 服务器落盘的命令日志（JSONL）意味着
   `MatchConfig + 命令序列 = 整局游戏`。断线重连靠 `resume_token` 重新握手、
   重挂席位、补发快照；录像/观战/回查都是同一套地基。

配套机制：回合计时 90s（超时代打 EndTurn，连续两回合超时判负）、掉线宽限 120s
（超时判 abandon 负）、`/healthz` 探活。

## 2. 数据存在哪、如何继承

### 客户端（每台设备）

Godot `user://` 目录，Windows 上即：

```
%APPDATA%\Godot\app_userdata\Hold the Line\
```

| 文件 | 内容 | 代码 |
|---|---|---|
| `identity.json` | 游客身份：`guest_id` + 随机 `secret`（首启生成） | `game/scripts/Identity.cs` |
| `decks.json` | 本地卡组库；每副同步到服务器后记 `ServerId` | `game/scripts/DeckStorage.cs` |

### 服务器（VPS）

SQLite 单文件 `/var/lib/holdtheline/holdtheline.db`（`holdtheline.service` 的
`HTL_Server__DbPath` 指定；**不设此变量则退化为内存库，重启全丢**）。存账号、云端卡组、
收藏、天梯 ELO。旁边 `match-logs/` 是对局命令日志。B2 起有每日在线备份。

### 继承规则

| 场景 | 要做什么 |
|---|---|
| 游戏版本更新（换 exe/zip） | 不用做。`user://` 在 AppData，与安装目录无关 |
| 关游戏重进 | 不用做，自动读 `identity.json` 重新握手 |
| 服务器重部署 | 不用做。`deploy.ps1` 只覆盖 `/opt/holdtheline`，数据库在 `/var/lib/holdtheline` |
| 换电脑/重装系统 | 拷贝上面的 AppData 目录（至少 `identity.json`）。丢了 = 新玩家（游客制下无法找回；v5 起有用户名密码账号可跨机登录） |

易混点：握手校验 **DataHash**（卡表/规则数据 SHA256）。客户端与服务器版本不一致会拒绝连接
（`data_mismatch`），但**不丢数据**——两端更到同版本即可用原身份继续。这也是
"协议 bump 后客户端与服务器必须同发"部署口径的由来（docs/00）。

## 3. 账号体系：从"无感游客"到"用户名密码"

### M3 游客制（协议 v2–v4）："本机密钥 = 身份"

界面上没有注册/登录，但协议层两者都在发生，只是全自动：

- **注册 = 第一次连服务器**。首启生成 `guest-xxxxxx` + secret；服务器首见该 `guest_id`
  即 INSERT（`AccountStore.RegisterOrRestore` → `Registered`），玩家无感。
- **登录 = 之后每次连接**。hello 带 guest_id + secret，服务器比对 secret 的 SHA256
  （恒定时间比较）；对不上拒绝握手（`bad_identity`），所以伪造 guest_id 冒充不了别人。

| 传统账号 | 游客制 |
|---|---|
| 填表点"注册" | 首启自动生成 identity.json |
| 输密码"登录" | 每次连接自动带 secret 握手 |
| 密码哈希存库 | secret 的 SHA256 存库 |
| 忘记密码可找回 | 没有找回——文件丢了就是新玩家 |
| 跨设备登录 | 不支持——身份绑在文件上 |

后两行是刻意接受的代价：朋友间封闭 Beta 不需要邮箱/防盗号，换来**零注册摩擦**
（解压即玩，一个输入框都不用填）。

### docs/12 B1（协议 v5）：用户名 + 密码

加入 `Register/Login/AuthOk` 消息，正式账号可跨设备登录、继承战绩卡组。
**正是这一步触发了 wss/TLS 的硬需求**（见下节）。

## 4. 为什么账号系统一来就要 wss/TLS

一句话：**B1 之后，线路上第一次出现了"真正值钱的秘密"——用户自己设的密码。**

之前 `ws://212.64.21.174:5210/ws` 是明文 WebSocket：路径上的任何一环（Wi-Fi、运营商、
中间路由）都能读到裸 JSON。当时可接受，因为线上唯一的秘密是随机生成的游客 secret：
不是人取的、不会复用、泄露最坏后果 = 有人冒充一个可抛弃的游客身份。

密码性质完全不同：**人类会跨网站复用密码**。朋友注册时填的可能是他银行也在用的密码——
让它以明文跑公网就不是游戏问题而是安全事故。行业铁律：**传输密码，传输层必须加密**。
wss = WebSocket over TLS，即给 ws 通道套上与 https 同级的加密。

TLS 还顺带解决**身份验证**：明文时代理论上可被中间人冒充服务器；有证书后客户端能验明
连的确实是 `htl.cicala.chat`。

### 连锁需求逐条解释

| 步骤 | 原因 |
|---|---|
| DNS A 记录 `htl` → 212.64.21.174 | TLS 证书签给**域名**而非 IP（Let's Encrypt 不给裸 IP 签）。副产品：客户端写死域名，将来换服务器只改 DNS |
| 防火墙放行 443 | wss 的实际入口（标准 HTTPS 端口） |
| 防火墙放行 80 | certbot 的 HTTP-01 域名验证要从公网访问 80 |
| nginx 反代 | 真正做 TLS 加解密的是 nginx，游戏服务器代码不碰证书（见下节） |

### 与裸 5210 的对比

```
之前 (M2~v4):
客户端 ──ws:// 明文──────────────────► 212.64.21.174:5210 (Kestrel 直接暴露公网)

现在 (v5, B0 之后):
客户端 ──wss:// 加密──► 443 nginx ──明文(仅本机内)──► 127.0.0.1:5210 Kestrel
                        (TLS 终止)
```

| | 裸 5210 | wss 反代 |
|---|---|---|
| 线路加密 | 无 | TLS |
| 客户端地址 | 写死 IP | 域名，可迁移 |
| 传密码 | 不可接受 | 可以 |
| 公网暴露面 | 游戏服务器直接暴露 | 只暴露 nginx |
| 防中间人 | 无 | 证书校验 |
| 网络兼容性 | 非标端口可能被公司/校园防火墙掐 | 443 几乎处处放行 |

**收口顺序（B2.3）**：现网 5210 仍对公网开着是刻意的——wss 验证通过前不能断现网。
稳定后再改 Kestrel 只听 `127.0.0.1:5210` + 安全组移除 5210，公网入口只剩 nginx。

## 5. nginx 与反向代理

**nginx 是什么**：开源的"网络流量门房"，两个角色——① Web 服务器（直接回静态文件，
Portcialio 就是这么伺候的）；② 反向代理（把请求转发给背后真正干活的程序）。

**约束在端口，不在 nginx**：一台机器一个端口同一时刻只能被一个进程监听，而 https/wss
的标准端口就是 443。这台 VPS 的 80/443 已被现有 nginx 占用，Kestrel 不可能再监听 443。
可选项只有两个：自用裸端口（旧 5210 方案）或加入 nginx 共享 443（现方案）——
从来不存在"独占"选项。

**Portcialio/PodEcho 能共存，靠的正是同一套机制**——虚拟主机（virtual host）：
客户端 https 请求带着目标域名，nginx 按域名把同一个 443 的流量分拣给不同后端。

```
                 公网 443 (唯一入口)
                      │
                   nginx  ← 按域名(Host)分拣
      ┌───────────────┼──────────────────┐
 www.cicala.chat   podecho.…          htl.cicala.chat
      │               │                   │
 直接回静态文件    转发本机端口上的       转发 127.0.0.1:5210
 (Portcialio)    Node 进程(PodEcho)     的 Kestrel(游戏服)
```

PodEcho 的 Node 进程也在本机高位端口裸跑、从不直接见公网——和 Kestrel 现在的姿势
完全一样。我们不是"打破了共享"，而是**加入了它们一直在用的共享机制**。

**反代为本项目解决的四件事**：

1. **端口共享**：一个 443 服务 N 个项目，按域名分流；玩家不用记非标端口。
2. **TLS 集中处理**（本次核心动机）：加解密、证书、certbot 90 天自动续期全在 nginx 一处；
   游戏服务器收到的已是解密后的普通 WS。若让 Kestrel 自己做 TLS：证书要喂给游戏进程、
   续期要写钩子重启、certbot 验证还是得占 80 找 nginx——绕不开，不如加入。
3. **不打扰现有项目**：往 `conf.d/` 加一个独立 `htl.conf` + reload（不停机），
   现有站点零改动。这也是不装 Caddy 的原因——Caddy 也要 80/443，会与现有 nginx 打架。
4. **缩小暴露面**：B2.3 后游戏服只听 127.0.0.1，公网唯一的门是 nginx；限流、日志、
   IP 封禁都有集中位置（`X-Forwarded-For` 即为把真实玩家 IP 透传给游戏服做限流）。

名词由来助记：**正向**代理替**客户端**出门（服务器不知道真实用户）；**反向**代理替
**服务器**看门（客户端不知道背后是哪个进程）。nginx 站在服务器一侧，故曰反向。
