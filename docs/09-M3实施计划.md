## 1. M3 的定义

**一句话目标:朋友圈封闭 Beta ——固定身份、自组卡组、排队匹配、天梯排名,卡池 ~160 张带正式美术。**

### 1.1 In(必须交付)

- **持久化与账号**:游客身份跨会话固定(本机密钥),昵称、卡组、战绩、排名入库(服务器 SQLite)。
- **卡组编辑器**:客户端收藏浏览 + 组卡 UI;服务端存卡组、`DeckValidator` 校验(已有,直接用)。
- **收藏数据模型**:每账号拥有卡表;**Beta 期配置为全解锁**(合成/尘经济只建模型不开放,D-M3-2)。
- **匹配队列 + 天梯**:单队列、ELO 评分、赛季表;匹配成功自动开局(复用 Room/MatchSession)。
- **规则型 AI**:GreedyAi 升级为 1~2 层搜索(SearchAi),用于人机模式与队列保底陪练。
- **卡池 ~160**:4 阵营 × ~30 + 中立 ~40(内容轨负责,见 §7)。
- **正式美术批量接入**:全卡插画/立牌 + 界面成品化(管线已跑通,批量执行)。
- **M2 遗留吸收**:终局同房「再来一局」;预结算(视 M2 DoD 延迟结论,条件项 D-M3-3)。

### 1.2 Out(明确不做,防蔓延)

- 密码/第三方登录、邮箱验证(M4;Beta 用"本机密钥 = 身份",丢了就是新号,朋友圈可接受)。
- 合成/尘经济的**开放**(模型入库但 Beta 全解锁;经济数值调优是上线前的事)。
- 商店、任务、成就、观战/回放 UI、好友列表(M4+)。
- Postgres/Redis 迁移(SQLite 在 Beta 规模绰绰有余,见 §3;仓库层留好接缝)。
- MessagePack、Steam 传输(接缝已留,M4 平台期)。

### 1.3 M3 要回答的问题

1. 自组卡组后,四阵营对局环境是否健康(没有一家独大 / 一套无解)?
2. 排队—对局—涨分的循环,能不能让试玩者**主动回来打第二天**?
3. 预结算是否必要(M2 DoD 的延迟体感结论在此落地)?

## 3. 架构增量

### 3.1 持久化:服务器 SQLite(决策 D-M3-1,默认已定)

与 PodEcho 同机同模式(它已在这台 VPS 用 SQLite 稳定运行)。Beta 规模(≤几十人,回合制)
远够;仓库接口留出,后续换 Postgres 不动业务代码。

```
src/HoldTheLine.Server/Data/
├── Db.cs                # Microsoft.Data.Sqlite;启动时幂等建表(CREATE TABLE IF NOT EXISTS)
├── AccountStore.cs      # accounts(guest_id PK, secret_hash, name, created_at, last_seen)
├── DeckStore.cs         # decks(id, guest_id, name, faction, cards_json, updated_at)
├── CollectionStore.cs   # collections(guest_id, card_id, count) —— Beta 全解锁时虚拟返回满额
└── LadderStore.cs       # ratings(guest_id, season, rating, wins, losses)
                         # match_history(id, season, guest0, guest1, winner_seat, reason, ended_at)
```

- 数据库文件:`/var/lib/holdtheline/holdtheline.db`(systemd 单元加一行 env)。
- 所有写操作走单写者队列(SQLite 单写者特性;参考 MatchSession 的 Channel 泵模式)。
- 备份:`match-logs` 同目录,cron 每日 `sqlite3 .backup`(部署脚本附带,一行)。

### 3.2 身份:持久游客

- 客户端首次启动生成 `guest_id` + 随机 `secret`,存 `user://identity.json`;hello 带两者。
- 服务器首见即注册(secret 哈希入库),后续 hello 校验 secret → 恢复身份。
- **协议 v2**:`Hello` 加 `secret` 字段;新增消息见 §3.4。旧版本号闸门直接挡掉 v1 客户端。

### 3.3 匹配与天梯

- `QueueManager`(服务器单例):`join_queue` 入队;配对策略 = 评分最近优先,等待越久容差越大;
  配上 → 内部创建 Room + MatchSession(复用 §2 全套)→ 双方收 `match_started`。
- 评分:ELO(K=32,新号前 10 局 K=64);赛季 = ratings 表的 season 字段,换季只加行不删档。
- 天梯榜:`get_ladder` 返回 top50 + 自己的名次。
- **队列保底**(D-M3-4):排队超 45s 无人 → 服务器起 SearchAi 机器人对局(明确标注"练习赛",
  不计入天梯,只计普通战绩)。Beta 人少时保证"点了必有局打"。

### 3.4 协议 v2 新增消息(一次性列全,实现中再增删 = 回 Fable,沿用 S2 纪律)

| 方向 | 消息 | 载荷 | 说明 |
|---|---|---|---|
| C→S | `save_deck` | deck_id?, name, card_ids | 建/改卡组(服务端 DeckValidator 校验) |
| C→S | `delete_deck` | deck_id | |
| C→S | `get_profile` | — | 要账号概况 |
| C→S | `join_queue` / `leave_queue` | deck_id | 排队/取消 |
| C→S | `get_ladder` | season? | |
| C→S | `rematch` | — | 终局后双方都发 → 同房重开(M2 遗留) |
| S→C | `profile` | name, rating, wins/losses, decks[], collection_mode | hello_ok 后自动推一次 |
| S→C | `deck_saved` / `deck_error` | deck_id / code+message | |
| S→C | `queue_status` | position?, waited_seconds, bot_fallback_in? | 每 5s 推 |
| S→C | `ladder` | entries[], my_rank | |
| S→C | `rematch_status` | you_ready, opponent_ready | |
| S→C | `rating_change` | old, new, season | match_ended 之后推 |

### 3.5 SearchAi(规则型 AI)

- 在 `HoldTheLine.Rules.Ai` 加 `SearchAi`:对每个合法命令用 Resolver 真实推演(状态本就不可变,
  天然支持),得子状态后**再展开一层己方最优应对**,评估函数复用 GreedyAi 打分并补齐:
  领袖血差、压力潮汐距离、守护覆盖、射程单位安全度。深度 2、每层取 top-8 剪枝,
  单步预算 ≤200ms(服务器陪练)/ ≤500ms(本地人机)。
- 验收基准:SearchAi vs GreedyAi 胜率 ≥65%(Sim 千局);四阵营互打仍在 40–60% 带内。

### 3.6 客户端:卡组编辑器 + 联机大厅改造

- **主菜单改造**:登录态显示昵称/评分;入口变为「排位匹配 / 人机练习 / 好友房间 / 卡组 / 热座」。
- **卡组编辑器场景**(新 `Deck.tscn`):左侧收藏网格(按阵营/费用筛选,复用卡面渲染与检视面板),
  右侧当前卡组列表(30 张进度、费用曲线柱状图、DeckValidator 实时红字);保存走 `save_deck`。
- **排队 UI**:队列状态浮层(已等待 Ns / 保底倒计时);匹配成功直接进 Battle。
- **终局**:「再来一局」(双方 `rematch` 就绪则同房重开)+ `rating_change` 展示 ±分。

## 4. 测试策略(在 M2 的 14 条上叠加)

1. **存储单测**:每个 Store 幂等建表、读写往返、并发写不丢(单写者队列)。
2. **协议 v2 往返** + 未知消息容错回归。
3. **集成(重头,仍走真 WebSocket)**:
   - 注册→存卡组(非法卡组被拒:31张/超稀有度上限/跨阵营)→ 排队→两客户端配对→打完→
     双方 `rating_change` 一致且 ELO 数学正确;
   - 断线重连后队列/对局状态正确;rematch 双就绪同房重开、单方退出则取消;
   - 队列保底:单人排队超时 → 与 bot 开局 → 不计天梯;
   - **服务器重启恢复**:重启后账号/卡组/评分还在(对局不恢复,与 M2 语义一致)。
4. **SearchAi 基准**:Sim 千局 vs GreedyAi ≥65%;耗时预算断言(单步 <500ms)。
5. **MCP 实测**:组一套非法卡组看红字→改合法→保存→排队→(双窗口或 bot)打完→看天梯名次。

## 8. 阶段计划(交 Opus,每阶段测试全绿 + 提交)

### B0 — 持久化与账号(0.5~1 周)
- SQLite 层 + AccountStore;hello 带 secret 的注册/恢复;`profile` 推送;
  客户端 `user://identity.json`;协议 v2 版本号切换。
- 顺手项:hello 加 **`data_hash`(卡表数据内容哈希)**,与服务器不一致时拒绝并提示更新
  ——补上"版本闸门只保护代码不保护数据"的缺口(M2 已知,改数值忘同步会静默劈叉)。
- **验收**:重启服务器身份仍在(集成测试);两台客户端各自稳定身份;数据哈希不匹配被拒。

### B1 — 卡组存取 + 编辑器(1~1.5 周)
- DeckStore + save/delete_deck + 校验错误回传;收藏(全解锁模式);
- 客户端卡组编辑器场景全套(§3.6);联机入口改为选"我的卡组"。
- **验收**:§4-3 卡组非法三案 + MCP 实测组卡→保存→用自组卡组开局。

### B2 — 匹配队列 + ELO(1 周)
- QueueManager、配对、`queue_status`、ELO 结算、`rating_change`、match_history;
- 大厅排队 UI;终局 rematch(同房重开)。
- **验收**:两客户端排队配对打完分数变动一致;rematch 双就绪重开。

### B3 — SearchAi + 队列保底(1 周)
- SearchAi(§3.5)替换人机模式;队列 45s 保底 bot(练习赛标注);
- Sim 基准跑分入 CI 惯例。
- **验收**:≥65% vs GreedyAi;单人排队保底可玩;人机模式体感"不送"。

### B4 — 天梯 + 运维 + Beta 开闸(0.5~1 周)
- `get_ladder`/榜单 UI/我的名次;healthz JSON;备份 cron;
- 部署到 VPS,发新 Windows 构建给试玩群 → **封闭 Beta 开始**。
- **验收**:§4-3 全绿;真人 ≥4 人各打 ≥3 局排位,榜单正确。

### B5 — 内容汇合 + 美术终版 + 打磨(1~1.5 周,与 Beta 运行重叠)
- 第二批卡入库后的适配(编辑器筛选、AI 评估权重复查);全卡美术批量接入;
- 预结算(若 D-M3-3 判需要);Beta 反馈的体验修补。
- **验收**:~160 卡全美术;Beta 周留存反馈回答 §1.3 三问。

**总计约 5~6.5 周**(不含内容轨自身工期;两轨并行)。风险最大项:B1 卡组编辑器
(纯新 UI 场景,工作量最大)与 SearchAi 的耗时预算(剪枝调不好会卡回合)——
B1 超期优先砍收藏筛选的花活,B3 超期允许退回深度 1 + 更强评估函数。
