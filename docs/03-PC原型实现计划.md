# 《守线 / Hold the Line》PC 原型实现计划

> 版本: v0.1
> 日期: 2026-07-16
> 前置文档:《01-游戏设计文档-GDD》(以 2026-07-16 修订版为准)、《02-技术PRD》
> 目标:用 Godot 4 + C# 做出一个**单机可玩的 PC 端原型**,具备核心玩法与基本体验循环,用于验证"好不好玩"并作为后续联机版本的地基。

---

## 1. 原型的定义:做什么,不做什么

### 1.1 原型要回答的三个问题

1. **核心循环好玩吗?** 部署→推进→卡位→破线的攻防节奏是否成立(GDD 支柱 P1/P2)。
2. **卡牌爽感还在吗?** 加了棋盘之后,打卡的节奏感是否被拖累(P3)。
3. **单局时长达标吗?** 8~12 分钟能否打完(P4)。

### 1.2 范围(In / Out)

| 在范围内(In) | 不在范围内(Out) |
|---------------|------------------|
| 5×4 棋盘完整规则(部署/移动/近战/远程/攻本体) | 联机(架构上预留,见 §3) |
| 交替回合制 + 费用系统 + 军令硬币 | 账号、收藏、合成、商店 |
| 10 个关键词(见 §4.2 分级)+ 战吼/亡语 | 地形格、装备卡、领袖阵营被动 |
| 2 个阵营 + 中立,共 **48 张卡**(见 §5) | 全部 4 阵营与 160 张卡池 |
| 人机对战(启发式 AI)+ 同屏热座 | 天梯、匹配、观战 |
| 基本体验循环:主菜单 → 选卡组(预组) → 对局 → 结算 → 再来一局 | 卡组编辑器(用预组卡组代替) |
| 疲劳机制(牌库抽干递增伤害) | 压力潮汐(GDD 已标记先不做) |
| 联机地基五条硬约束(§3.1)+ 局域网冒烟验证(P5 弹性) | 完整联机(房间/匹配/断线重连/服务器部署) |
| AI 美术工作流 + 48 张卡首版 AI 插画(§9) | 人工美术、动画、音乐、卡牌语音 |
| 4 个核心音效(免费音效库) | |

> 取舍原则:**凡是不影响"这三个问题"答案的系统一律砍掉**。卡组编辑器、收藏等体验循环外围,等原型验证通过后在正式版本做。

## 3. 架构:原型期就按联机形态写

原型是单机的,但内部仍然走**命令→结算→事件**单向流(技术 PRD §4/§6),只是"服务器"是本地进程内的一个对象:

```
InputController ──Command──▶ LocalGameHost(持有规则引擎) ──Event[]──▶ EventPlayer(动画队列) ──▶ BattleView
      ▲                            │
   拖拽/点选                  AI 也走同一入口:AiPlayer 产出 Command
```

- `LocalGameHost` 接口设计成与未来的 `NetworkGameHost` 同型(`SubmitCommand(cmd) / OnEvents(handler)`),联机时替换宿主即可,表现层零改动。
- **表现与结算严格分离**:BattleView 不读规则、不算伤害,只消费事件播动画。这条纪律在原型期就立起来,是这份计划里最重要的工程约束。
- 隐藏信息纪律同样保留:表现层只拿"对手手牌数"而非牌面(伏兵关键词也依赖这一层)。

### 3.1 联机地基:原型期强制执行的五条硬约束

"之后改联机改动会不会太多"取决于原型期有没有偷懒走捷径。以下五条从 P0 开始强制执行,并用测试/代码审查把关——做到了,联机化就只是**新增一个传输层和一个服务器进程壳**,规则引擎与整个对战表现层一行不改:

| # | 硬约束 | 把关方式 |
|---|--------|----------|
| 1 | **Command/Event 全部可序列化**(JSON 起步,预留 MessagePack),字段带版本号 | P0 起对每个类型做序列化往返(round-trip)单元测试 |
| 2 | **回环序列化模式**:Debug 构建下,即使单机,每个命令/事件也强制走"序列化→反序列化"再使用 | 防止"引用共享""不可序列化对象"等单机捷径潜入;CI 全程开启 |
| 3 | **隐藏信息在宿主层裁剪**:宿主对每个座位输出独立的视角事件流(对手手牌只有张数),表现层拿不到全量 GameState | 静态规则:`game/scripts/` 禁止引用 `GameState`/`Resolver`,只允许引用事件与视图 DTO;代码审查 + 架构测试 |
| 4 | **确定性 + 命令日志**:RNG 种子化,每局落盘"初始状态 + 命令序列",任意一局可离线重放 | 重放测试进 CI(跑一局录像,重放后终局状态哈希一致)。这同时就是联机的断线重连/对账机制 |
| 5 | **异步接口形态**:`IGameHost.SubmitCommand()` 返回可等待结果、事件经由回调异步到达——单机下也不允许"同步调用直接拿返回值"的写法 | 接口签名本身杜绝;AI 玩家同样只通过 IGameHost 交互 |

> 额外的低成本保险(P5 弹性项):写一个最薄的 WebSocket 宿主壳,两台机器局域网直连打一局作为**联机冒烟验证**——不做房间、匹配、断线重连,只证明"宿主可替换"这个假设成立。约 2~3 天工作量,建议保留。

## 4. 规则引擎范围

### 4.1 核心规则(全部实现)

- 5×4 棋盘;部署限己方底线行空格;集结中(召唤失调)。
- 回合流:费用+1(上限10)/恢复/抽牌 → 主要阶段 → 回合结束;先手后手(后手多 1 牌 + 军令硬币,含"+1费 / +1移动"二选一)。
- 移动:上下左右 1 格、不可穿越;每随从每回合 1 移动 + 1 攻击。
- 战斗:近战互殴;射程 N 直线无阻挡、不吃反击;底线行攻击领袖。
- 胜负:领袖 25 HP 归零;疲劳(抽干后 1,2,3… 递增伤害);投降。

### 4.2 关键词分两批(按修订版 GDD 的 12 个词条)

| 批次 | 关键词 | 说明 |
|------|--------|------|
| **第一批(P2 阶段)** | 冲锋、突袭、疾行 N、射程 N、守护、坚守、驻防、践踏、偷袭、持盾 + 战吼/亡语 | 全部是确定性结算,实现直接 |
| **第二批(视进度,可滑出原型)** | 伏兵 | 涉及隐藏信息的 UI 表达与 AI 处理,复杂度最高;砍掉不影响原型三问 |

### 4.3 效果原语(支撑 48 张卡即可)

原语:`damage / heal / buff / summon / draw / destroy / move_unit / grant_keyword`(8 个);
选择器:`self / target_unit / adjacent / row / column / half_board / cell / random_enemy_in_row`(8 个)。
超出组合能力的卡在原型期**直接不设计**,不写专属脚本,控制复杂度。

## 5. 卡牌内容范围:2 阵营 × 48 张

选**铁誓军团(墙)**与**荒野游群(风)**——攻防两极,是检验"推线张力"的最佳对照组(焰、械涉及行列 AOE 与建筑,系统成本高,留正式版)。

| 池 | 数量 | 构成 |
|----|------|------|
| 中立 | 16 张 | 白板身材曲线 1~7 费 + 基础功能卡(抽牌/小硬移除) |
| 铁誓军团 | 16 张 | 守护/坚守/驻防密度高,1~2 张持盾,慢曲线 |
| 荒野游群 | 16 张 | 冲锋/疾行/偷袭密度高,快曲线 |
| 领袖 | 2 名 | 铁誓「筑垒」/ 游群「狩猎号角」(GDD §5 技能原文) |

- 张数上限按修订版 GDD:普通 4 / 稀有 3 / 史诗 2 / 传说 1。
- 预组卡组 2 套(每阵营一套,30 张),玩家和 AI 都从这两套里选。
- 指令卡占比约 30%,至少各 1 张"以列为目标"的卡,专门验证空间法术的手感。

## 6. AI 对手

原型必须单人可玩,AI 是体验循环的一环,但只做**启发式一层评估**,不做搜索:

1. 枚举当前所有合法命令(规则引擎天然提供合法性校验,枚举是免费的);
2. 每个命令用评分函数打分:身材交换差 × 推进距离 × 底线威胁 × 保脸权重;
3. 贪心执行,重复到无可用行动后结束回合。

- 目标强度:**能惩罚玩家的明显失误,能赢新手** 即可;不追求强度。
- 附带产出:`HoldTheLine.Sim` 让两个 AI 互打 1000 局,输出先手胜率、平均回合数、每张卡出场/胜率表——这是数值调整的仪表盘,比手感争论高效得多。

## 7. 体验循环(原型版)

```
主菜单 ──▶ 选卡组(铁誓/游群 二选一,同时选对手AI卡组) ──▶ 对局 ──▶ 结算画面(胜负/回合数/本局伤害统计) ──▶ 再来一局 / 回主菜单
```

- 结算画面带**本局统计**(总回合、造成伤害、破线次数)——原型期这就是"奖励感"的替代品,也顺便帮我们收集验证数据。
- 热座模式(同机双人)藏在主菜单二级入口,用于线下面对面测试。
- 设置页只做:音量、全屏切换、回合计时开关(测试期默认关)。

## 8. 表现层要求(占位美术下的手感底线)

占位美术 ≠ 占位手感。原型要验证"爽不爽",以下五件事必须做到位:

1. **合法位高亮**:拖起卡/选中随从时,可部署格/可移动格/可攻击目标即时高亮;
2. **拖拽跟手**:卡牌跟随指针带缓动倾斜,松手吸附格子;
3. **攻击顶撞**:攻击方向目标顶撞位移 + 受击闪白 + 伤害数字弹出;
4. **事件队列节奏**:亡语连锁按 0.3~0.5s 间隔逐个播,不糊成一团;
5. **四个核心音效**:出牌、移动落子、攻击命中、破线打脸(免费音效库即可)。

占位视觉方案:纯色棋盘 + 色块立牌(阵营色区分,数字直接印在立牌上)+ 文字卡面。我可以用简单几何风格直接生成,不阻塞开发。

## 9. AI 美术生成方案与工作流

美术全部由 AI 生成。技术 PRD 选定的"桌面战棋质感(立牌棋子)"方向恰好是对 AI 生成**最友好**的美术方案:每张卡只需一张静态插画,立牌由插画裁剪而来,无需动画、无需拆件绑骨——这两项正是当前 AI 美术的短板。

### 9.1 资产清单与规格

| 资产 | 数量(原型) | 规格 | 来源 |
|------|-------------|------|------|
| 卡面插画 | 48 张 | 1024×1024 生成 → 裁切 4:3 入框 | AI 生成 |
| 场上立牌 | 48 个 | 同插画主体去背裁剪 + 统一底座 | 插画自动后处理,**不单独生成** |
| 领袖头像 | 2 张 | 512×512 圆形裁切 | AI 生成 |
| 棋盘 | 1 张 | 战地作战桌质感,格线由程序绘制叠加 | AI 生成底图 + 程序格线 |
| 卡框/UI 组件 | 1 套 | 阵营色双套卡框、费用/攻/血宝石、按钮 | AI 生成元素 + 手工排版(UI 是 AI 弱项,几何为主) |

### 9.2 风格一致性:三层防线

AI 批量出图的头号风险是"48 张卡像 48 个游戏"。对策从强到弱:

1. **风格圣经(Style Bible)**:一份入库的提示词规范文件——固定的风格描述块(战地手账、油画笔触、冷灰暖褐 + 辉尘青金)、四阵营各自的主色与元素词表、统一的负面提示词、统一的构图规则(单主体、3/4 视角、背景虚化以便去背)。**每张卡的提示词 = 风格块 + 阵营块 + 该卡专属描述**,专属部分越短,风格越稳。
2. **风格参考图驱动**:选定 3~5 张"锚点图"后,所有生成都挂风格参考(image prompt / IP-Adapter / style reference,视工具而定),而不是纯文字。
3. **风格 LoRA(可选,正式版再上)**:用首批验收通过的 30~50 张图训练专属风格 LoRA,批量生产期一致性最高。原型期 48 张靠前两层足够,不为原型引入训练环节。

### 9.3 工具链选型 【待定夺】

| 方案 | 组成 | 优点 | 缺点 | 适合 |
|------|------|------|------|------|
| **A. 云端 API 全流程(推荐起步)** | 高质量图像模型 API(如 gpt-image / Imagen / 即梦等可编程接口)+ 脚本批量调用 | 零环境搭建;我可以直接写脚本驱动全流程(生成→筛选→后处理);按张付费,48 张成本极低 | 风格控制手段少于本地方案;依赖服务商 | 原型期,快速验证 |
| B. 本地 ComfyUI + Flux/SDXL | 本地工作流 + IP-Adapter/ControlNet + 自训 LoRA | 可复现(workflow JSON 入库)、免按张付费、风格控制上限最高 | 需 ≥12GB 显存的 GPU;搭建与调参有学习成本 | 正式版批量生产 |
| C. 订阅制工具(Midjourney 等) | 网页/Discord 手动出图 | 出图美感上限高 | **无法脚本化**,人肉工作流与"AI 全自动生成"目标相悖 | 仅用于风格探索找感觉 |

**推荐路线:A 起步,验证风格后视情况迁移 B。** 你的显卡配置会影响这个决策,开工前告诉我。

### 9.4 自动化流水线(脚本化,入库)

```
cards/*.json (卡牌数据)
   │  提示词生成器:风格圣经 + 阵营块 + 卡牌描述字段 → prompt
   ▼
生成脚本(每卡出 4 候选)──▶ 人工筛选(你,唯一人肉环节:选 1/4 或打回重roll)
   ▼
后处理脚本(全自动):去背(rembg)→ 主体裁剪出立牌 + 统一底座合成
   → 卡面裁切入框 → 尺寸/命名规范化 → 输出到 game/assets/cards/
   ▼
Godot 按卡牌 ID 约定自动加载(缺图自动回退几何占位,美术永不阻塞开发)
```

- **提示词与卡牌数据同库**:每张卡的 JSON 里直接带 `art_prompt` 字段,美术可随时重新生成,提示词改动有 Git 历史。
- **人工环节只留一个**:四选一筛选。按 48 张 × 平均 1.5 轮估算,你的总投入约 2~3 小时。

### 9.5 与阶段计划的挂钩

| 时间点 | 美术动作 |
|--------|----------|
| P0~P2 | 全部几何占位,零美术投入;`art_prompt` 字段随卡表一起写好 |
| P3 期间(并行) | **风格探索包**:1 张棋盘 + 4 张卡面 + 2 个立牌,定风格圣经 v1;搭好 §9.4 流水线脚本 |
| P4 期间(并行) | 48 张卡批量生成与筛选 |
| P5 | 全量替换占位,整体审一遍色调统一性 |

## 10. 阶段计划

以"我(AI)承担主要编码 + 你负责试玩/拍板/调数值"的协作模式估算。每阶段末尾有**可验收产物**,你每周都摸得到新东西。

### P0 — 工程脚手架(0.5~1 周)
- Git 仓库、解决方案结构(§2.1)、Godot 工程初始化、xUnit/GdUnit4 跑通、卡牌 JSON Schema 与加载器。
- 联机地基落地(§3.1):IGameHost 异步接口、Command/Event 序列化框架 + 往返测试、回环序列化开关、表现层依赖隔离的架构测试。
- **验收**:空工程可运行,`dotnet test` 绿灯(含序列化往返测试),CI 可选。

### P1 — 规则引擎核心(1.5 周)
- §4.1 全部核心规则 + 命令/事件模型 + 状态快照;白板卡 16 张(中立池)。
- xUnit 覆盖:每条规则至少一测,重点覆盖移动阻挡、射程遮挡、集结中、疲劳。
- **验收**:命令行文本模式可打完一整局(无 UI);测试 ≥ 60 条全绿。

### P2 — 关键词与卡池(1.5 周)
- 效果原语 + 触发器总线;第一批 10 关键词 + 战吼/亡语;48 张卡全部进 JSON。
- `HoldTheLine.Sim` 随机 AI 互打出第一份数值报表。
- **验收**:每个关键词、每张有效果的卡各有结算测试;模拟器报表产出。

### P3 — 对战场景(2 周)
- BattleView(棋盘/立牌/手牌/费用/领袖血量)、InputController(拖拽/点选)、EventPlayer(动画队列)、§8 五项手感底线。
- 先接热座模式(双人同机),因为它不依赖 AI。
- **验收**:两人一台机可打完整局,手感底线五项达标。⭐ **第一个"好不好玩"检查点**

### P4 — AI 与体验循环(1.5 周)
- 启发式 AI 接入 `LocalGameHost`;主菜单/选卡组/结算画面串成循环;设置页。
- **验收**:从双击 exe 到"再来一局"全流程无断点;AI 能赢乱打的玩家。

### P5 — 调优与验证(1~1.5 周,弹性)
- 你 + 找 3~5 个朋友试玩;结合模拟器报表调卡牌数值;修体验毛刺;Windows 导出 exe。
- AI 美术首版接入:48 张卡插画 + 立牌替换几何占位(工作流见 §9,风格探索在 P3 期间已并行完成)。
- 弹性项(按此优先级取舍):① 局域网 WebSocket 冒烟验证(§3.1);② 伏兵关键词。
- **验收**:**可分发的 Windows 构建**(带首版 AI 美术);10 局试玩数据回答 §1.1 三个问题。

> **总计约 8 周。** P1/P2(规则引擎)与 P3(表现层)之间有部分并行空间,顺利的话可压到 6~7 周。风险最大的是 P3 的手感打磨——预算不动,宁可砍伏兵和卡池张数。

## 11. 原型完成的判定(Definition of Done)

全部满足即宣告原型阶段结束,进入正式版决策:

- [ ] 双击 exe → 选卡组 → 对局(人机)→ 结算 → 再来一局,全程无崩溃;
- [ ] 单局中位时长落在 8~15 分钟(原型期放宽上限);
- [ ] 两套预组互打胜率在 40%~60% 区间(模拟器 + 实战);
- [ ] 试玩者(≥3 人)反馈中,"站位/推线有意思"为主流意见;
- [ ] 规则引擎测试 ≥ 150 条,全绿;表现层与规则层零耦合(BattleView 不引用 Resolver)。

据此做 **Go/No-Go**:Go → 按技术 PRD 的 M2 进入联机垂直切片;No-Go → 回炉核心规则(棋盘尺寸/行动点等 GDD §12 备选项)。

## 13. 模型分工与切换点(Fable 5 ⇄ Opus 4.8)

原则:**按决策密度分工**——语义与架构决策用 Fable 5,规格明确的批量实现用 Opus 4.8(可开 Fast 模式)。

### 13.1 分工表

| 工作内容 | 模型 | 说明 |
|----------|------|------|
| P0 脚手架 + 联机地基五约束落地 | **Fable 5** | 架构决策集中 |
| P1 规则引擎核心:状态/命令/事件模型、结算器、移动与战斗语义、宿主接口、触发器总线骨架 | **Fable 5** | 全项目正确性风险最高的代码 |

### 13.2 切换信号 S1(Fable → Opus)

以下三条**全部满足**即可切换,由 Fable 在会话中明确宣布:
