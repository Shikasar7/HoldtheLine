## 1. 产品决策(已定,不再讨论)

- **炉石式子集重抽**:双方看到各自起手后,可**勾选任意子集**换掉,**只有一次机会**;
  重抽张数 = 换掉的张数。
- **防重复顺序**:先从牌库顶抽新牌,**再**把被换的牌洗回牌库 —— 结构上保证换掉的牌不会被
  立刻抽回。
- **全平台**:单机(vs-AI / 热座)与联机都支持。

## 2. 设计总览(关键决策一览)

| # | 决策点 | 结论 | 理由 |
|---|---|---|---|
| D1 | mulligan 阶段建模 | `GameState.Mulligan`(可空的 `MulliganState`),`null` = 正常对局 | 不加新枚举字段;旧快照/旧日志反序列化默认 null,天然加性 |
| D2 | 玩家输入形态 | 新 `MulliganCommand{Seat, ReplacedEntityIds}`,走既有 `SubmitCommand` 通道 | 命令即日志:config + 命令日志 = 完整重放(硬约束 #4);**协议层零新增 C→S 消息** |
| D3 | 开关位置 | `MatchConfig.MulliganEnabled`,**默认 false**;生产入口显式置 true | 旧命令日志的 config JSON 无此字段 → 反序列化为 false → **旧日志重放不受影响**。默认 true 会让所有旧日志第一条命令被拒 |
| D4 | RNG 座位隔离 | `MulliganState` 内存两条独立 SplitMix64 流(由 `Seed ^ 座位盐` 派生),主 `state.Rng` **不被 mulligan 消耗** | 双方 mulligan 结果与提交顺序无关;对手无法从后续公共随机结果反推你换牌与否/张数影响 |
| D5 | 洗回算法 | 先按张数从牌库顶抽 k 张 → 被换的 k 张 append 进牌库 → 用**本座位** RNG 对整个牌库 `Shuffle` | 满足"先抽后洗";整库重洗实现最简、泄露最少、最易断言 |
| D6 | 被换牌的 EntityId | **保留原 id,不重铸** | 对手本来就能看到 CardDrawnEvent 里的 entity id;重铸会让"高位 id"暴露这张是洗回的牌(对手侧泄露)。保留原 id 只让持有者自己认出"抽回了换掉的那张"——己方信息,可接受 |
| D7 | 军令硬币 | 从 `CreateGame` 移到**双方 mulligan 完成后**发放(仅 MulliganEnabled 时) | 硬币不属于牌库,不可被换;发放后置则 UI 无需特判"这张不能勾" |
| D8 | 换牌张数是否公开 | 公开(对手可见 ReplacedCount) | 与炉石一致;约束只要求"换了什么/抽到什么"保密,张数是通用公共信息 |
| D9 | mulligan 计时 | 服务器一条共享倒计时(`ServerOptions.MulliganSeconds`,默认 45s);超时对未提交座位**自动"全留"**,不计 forfeit、不计 timeout streak | 开局阶段超时是良性的;回合双超时判负的严厉度不适用 |
| D10 | 版本 | `RulesInfo.Version` 0.3.0 → **0.4.0**;`ProtocolConstants.ProtocolVersion` 3 → **4** | 结算语义与协议语义都变了;版本闸门挡掉旧客户端 |
| D11 | 协议 S→C | **不加新消息**;`MatchStarted` / `ResyncOk` 各加可空 `MulliganSecondsLeft` 字段 | mulligan 每局只发生一次且时机固定在 match_started,倒计时随快照带下去即可(重连走 ResyncOk 同理) |

## 3. 规则层设计(HoldTheLine.Rules)

### 3.1 状态:`MulliganState`(新,入 `State/GameState.cs` 同文件)

```csharp
public sealed class MulliganState
{
    public bool[] Done { get; set; } = { false, false };   // 各座位是否已提交
    public ulong[] RngState { get; set; } = new ulong[2];  // 各座位独立 SplitMix64 流
}
// GameState 增: public MulliganState? Mulligan { get; set; }   // null = 不在 mulligan 阶段
```

- 派生:`RngState[s] = new DeterministicRng(config.Seed ^ (MulliganSalt + (ulong)s)).State`,
  `MulliganSalt` 为固定常量(如 `0x4D554C4C_49474E00UL`)。SplitMix64 不同种子即独立流,
  与主流零共享。
- `MulliganState` 随 GameState 序列化(Resolver 克隆语义、快照恢复都免费获得)。
- 阶段判定处处用 `state.Mulligan is not null`,完成时**置回 null**(状态回到今天的形状,
  后续序列化与旧版逐字段等价)。

### 3.2 命令:`MulliganCommand`(Commands/Command.cs)

```csharp
[JsonDerivedType(typeof(MulliganCommand), "mulligan")]   // 加入基类注册表
public sealed record MulliganCommand : Command
{
    /// <summary>要换掉的手牌 EntityId;空列表 = 全留。</summary>
    public required IReadOnlyList<int> ReplacedEntityIds { get; init; }
}
```

Resolver.Execute 校验链(在现有 seat 校验之后):

1. `state.Mulligan is not null` 时:除 `MulliganCommand` 与 `ConcedeCommand` 外一律拒绝
   (新错误码 `RuleErrorCode.MulliganPending`);`MulliganCommand` **绕过 ActiveSeat 检查**
   (与 Concede 同类,双方都可行动)。
2. `state.Mulligan is null` 时收到 `MulliganCommand` → 拒绝(`InvalidCommand`)。
3. 本座位 `Done[seat]` 已 true → 拒绝("只一次")。
4. `ReplacedEntityIds`:去重后必须全部在本座位手牌中;`Count ≤ Deck.Count`(小牌库测试
   场景防御,真实对局 24 vs ≤6 永不触发),否则拒绝。

### 3.3 结算:`ResolveMulligan`(Resolver 新方法)

```
seatRng = new DeterministicRng { State = mull.RngState[seat] }
1. 把 ReplacedEntityIds 对应的 CardInstance 从 Hand 取出、暂存 aside(不进事件)
2. 抽 k = aside.Count 张:直接从 Deck 尾部(牌库顶)取 → Hand,逐张 Emit CardDrawnEvent
   (不走 ctx.DrawCards —— 那条管线带疲劳/爆牌语义,mulligan 阶段不适用;张数已在校验期封顶)
3. aside 整批 append 回 Deck,seatRng.Shuffle(整个 Deck)
4. mull.RngState[seat] = seatRng.State(写回,虽然一次性用完,保持"RNG 即状态"的既有纪律)
5. Emit MulliganResolvedEvent{Seat, ReplacedEntityIds, ReplacedCount}
6. Done[seat] = true;若双方都 Done:
   a. state.Mulligan = null
   b. Emit MulliganCompletedEvent
   c. 发硬币(原 CreateGame 中的逻辑原样搬入:TakeEntityId + Hand.Add + CardDrawnEvent)
   d. TurnFlow.StartTurn(ctx, config.FirstSeat)   ← 见 3.4 的 FirstSeat 传递方式
```

- **顺序无关性**:两座位各用自己的流、只动自己的 Hand/Deck,先到先结算但结果与顺序无关
  (事件 Sequence 当然随日志顺序走,这不破坏重放)。
- **全留(k=0)**:跳过 1–4,只走 5–6。不消耗 RNG、不重洗 —— "什么都没发生"就真的什么都没发生。

### 3.4 `GameFactory.CreateGame` 拆分

`MulliganEnabled == false`:与今天逐字节相同(旧日志重放、现有测试全部不动)。

`MulliganEnabled == true` 时结尾改为:

```
发起手(DrawCards ×2,不变)
[删]发硬币(挪到 3.3-6c)
[删]TurnFlow.StartTurn(挪到 3.3-6d)
[加]state.Mulligan = new MulliganState{ RngState = 由 Seed 派生 }
```

FirstSeat / CoinCardId 在 3.3-6 需要用到而 Resolver 拿不到 MatchConfig →
把两者存进 `MulliganState`(`int FirstSeat; string CoinCardId;`),完成时读取。
(比给 Resolver 注入 MatchConfig 侵入小;GameState 本就带对局参数的既有先例是 RulesVersion。)

注意:mulligan 阶段 `TurnNumber == 0`、`ActiveSeat == FirstSeat`(维持现值,但**任何回合逻辑
都不会跑** —— StartTurn 未被调用,不加费、不抽回合牌)。

### 3.5 事件(Events/GameEvent.cs,加性扩展,Fable 审定即本文)

```csharp
[JsonDerivedType(typeof(MulliganResolvedEvent), "mulligan_resolved")]
[JsonDerivedType(typeof(MulliganCompletedEvent), "mulligan_completed")]

public sealed record MulliganResolvedEvent : GameEvent
{
    public required int Seat { get; init; }
    /// <summary>被换牌的 EntityId;对手视角裁剪为 null。</summary>
    public IReadOnlyList<int>? ReplacedEntityIds { get; init; }
    /// <summary>公开信息(D8):换了几张。</summary>
    public required int ReplacedCount { get; init; }
    public override GameEvent RedactFor(int viewerSeat) =>
        viewerSeat == Seat ? this : this with { ReplacedEntityIds = null };
}

public sealed record MulliganCompletedEvent : GameEvent;   // 无载荷:阶段结束的显式信号
```

- 重抽的新牌复用 `CardDrawnEvent`(已有对手裁剪 CardId → null);**不需要**新的"洗回"事件
  ——持有者由 `ReplacedEntityIds` 得知移除哪几张,对手只该知道 `ReplacedCount`。
- 不加 "mulligan_started" 事件:客户端从 PlayerView 的 mulligan 标志(3.6)得知阶段,
  事件面收敛。

### 3.6 视图与枚举

- `PlayerView`(Hosting/PlayerView.cs)加性新增:
  - `public bool MulliganPending { get; init; }` —— **本座位**尚未提交(UI 开关);
  - `public bool OpponentMulliganPending { get; init; }` —— 对手尚未提交("等待对手…")。
  - `From()` 由 `state.Mulligan` 推导;旧协议对端反序列化缺字段 = false,无害。
- `CommandEnumerator.LegalCommands`:`state.Mulligan is not null` 时,对**每个未 Done 的座位**
  只返回一条 `MulliganCommand{ReplacedEntityIds=[]}`(全留)。不枚举 2^n 子集 —— 枚举器的
  用途是"随便挑一条都合法"(bot 保底/超时代打/随机 bot),启发式换牌由 AI 层自行构造(§7)。
  注意签名:现在按 `state.ActiveSeat` 枚举,需加可选 seat 参数或在 mulligan 分支内部改按
  传入 seat(LocalGameHost / MatchSession 调用点同步)。
- `LocalGameHost.LegalCommands / SuggestCommand`:现有 `seat != ActiveSeat → 空` 的闸门加
  mulligan 分支:阶段内未 Done 的座位返回上述全留命令 / AI 的 MulliganPick(§7)。

## 5. 服务器层(HoldTheLine.Server/Rooms/MatchSession.cs)

原则:mulligan 是"match_started 之后、第一个回合计时器之前"的阶段;**双方都完成后**才
StartTurn、才起 `_turnTimerCts`。

### 5.1 时序

```
Room/QueueManager: MatchSession.Create(config 置 MulliganEnabled=true)   ← 唯一生产入口,rematch 自动继承
SendMatchStartedAsync: 每座位的 MatchStarted 带 MulliganSecondsLeft = opts.MulliganSeconds
Begin(): 现在写 TurnBegin 信号 → 改为:host 处于 mulligan 阶段时改起 mulligan 计时器(5.2),
         否则维持现状(MulliganEnabled=false 的路径 / 未来复用)
双方 MulliganCommand 经 HandleSubmitAsync 正常结算(LogCommand 照旧 → 日志完整)
第二个 mulligan 的事件批自带 MulliganCompleted + CardDrawn(硬币) + TurnStarted
  → HandleSubmitAsync 尾部的 SyncTurnTimerAsync 看到 ActiveSeat 且阶段已结束 → 首个回合钟启动
```

### 5.2 具体改动

- `ServerOptions` 加 `MulliganSeconds`(默认 45;0 = 立即代打全留,供测试)。
- 新字段 `CancellationTokenSource? _mulliganCts` + 信号 `Signal.MulliganTimeout`:
  `Begin()` 时若 `Host` 在 mulligan 阶段(通过 `Host.GetView(0/1)` 的 mulligan 标志判断,
  服务器不触碰 GameState —— 现有纪律),起一次性 `RunMulliganTimerAsync`。
- `HandleMulliganTimeoutAsync`:对每个 `MulliganPending` 的座位,以其名义提交
  `MulliganCommand{ReplacedEntityIds=[]}`(照常 LogCommand → 重放含代打命令),随后走
  FanOut / CheckMatchEnd / SyncTurnTimer。**不碰 `_timeoutStreak`、不 forfeit**(D9)。
- `SyncTurnTimerAsync` 头部加早退:host 仍在 mulligan 阶段 → return(**必须按阶段判断,
  不能按 ActiveSeat** —— mulligan 期间 ActiveSeat 恒为 FirstSeat,不早退会把回合钟提前打开)。
- 双方都提交后(在 HandleSubmitAsync 里检测阶段结束)`_mulliganCts?.Cancel()`。
- **重连**:`HandleReattachAsync` 的 ResyncOk 加 `MulliganSecondsLeft`(用记下的 deadline 换算
  剩余秒)。掉线期间 mulligan 计时**不暂停**(超时=代打全留,损失有限;与回合钟"掉线不停表"
  的现有语义一致)。宽限/判负逻辑零改动(Concede 在 mulligan 阶段合法,§3.2-1)。
- **保底 bot 席**(D-M3-4,尚未实装):实装时 bot 驱动在 match_started 后立即提交
  `LegalCommands` 里的全留命令(或 §7 启发式)。本计划只约定接口,不阻塞。

## 6. 客户端层(接口约定,Opus 施工;不含 Godot 代码)

数据流(单机 LocalGameHost / 联机 RemoteGameHost 完全同构,BattleScene 只认 IGameHost):

1. **进场判定**:首个 `GetView(mySeat)`(联机即 match_started 快照)`MulliganPending == true`
   → 进入 mulligan 子状态,**不得**假设"马上有 TurnStarted"(现 BattleScene 若有此假设需解除
   —— 风险项 R4)。
2. **Mulligan 面板**:展示 `view.Self.Hand`(先手 4 张 / 后手 6 张,此时**没有硬币**,D7);
   每张牌可勾选"换掉";确认按钮 + 倒计时(联机用 `MulliganSecondsLeft` / 单机用本地配置,
   单机可无限时)。确认 → `SubmitCommandAsync(mySeat, new MulliganCommand{ReplacedEntityIds=勾选})`。
3. **等待态**:提交后若 `view.OpponentMulliganPending` → "等待对手换牌…";收到对手的
   `MulliganResolvedEvent`(裁剪版,含 ReplacedCount)可播"对手换了 N 张"提示。
4. **动画事件序**:己方 `MulliganResolvedEvent`(移除被换牌)→ `CardDrawnEvent ×k`(补入新牌)
  → (双方齐后)`MulliganCompletedEvent` → 硬币 `CardDrawnEvent`(后手)→ `TurnStartedEvent`
   → 关闭面板、走既有开局表现。
5. **热座**:FirstSeat 先做 mulligan → 既有换座遮罩 → 另一座位做 → 完成后进第一回合。
   两座位面板期间只渲染当前座位的 `GetView(seat)`(遮罩防偷看,复用现有热座交接约定)。
6. **vs-AI**:AI 座位在进场时用 `_localHost.SuggestCommand(aiSeat)`(已扩展,§3.6)自动提交;
   人类照 2。
7. **联机重连**:`ResyncOk.MulliganSecondsLeft != null` → 直接重建 2/3 的面板或等待态
   (由 view 两个标志区分);`RemoteGameHost.MulliganTimerReceived`/属性供倒计时。

## 7. AI / Sim

- `GreedyAi` 加 `public static MulliganCommand MulliganPick(GameState/PlayerView 手牌, CardDatabase db, int seat)`
  (放 Rules.Ai,单机与服务器 bot 共用):**v1 启发式 = 换掉所有 Cost ≥ 5 的牌**,其余全留
  (任务给的起步口径"换掉费用≥X";X=5 与 4/6 起手的费用曲线匹配,数值调优留给 Sim)。
  `SearchAi` 直接委托同函数 —— 隐藏信息下 rollout 无意义,不搜索。
- `LocalGameHost.SuggestCommand`:mulligan 阶段返回 `MulliganPick`。
- `Sim/Program.cs`:`PlayMatch` 的 config 置 `MulliganEnabled = true`(加 CLI 位控制,默认开,
  保证平衡口径 = 玩家实际体验);主循环前插:
  `while (state.Mulligan != null) 对每个未 Done 座位 resolver.Execute(MulliganPick(...))`。
  随机模式用枚举器的全留命令。跑通后**重跑 roundrobin 基准**留档(§9 M-5 验收)——
  mulligan 平滑曲线可能移动先手胜率(MatchConfig 注释记载的 ~57% 口径需复核)。
