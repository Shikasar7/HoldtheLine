## 1. 范围与技术目标

| 目标 | 指标 |
|------|------|
| 联机 1v1 对战稳定可用 | 断线重连 < 5s 恢复;服务端权威,客户端不可作弊 |
| 单局流畅 | 客户端 60 FPS;回合操作到对端呈现延迟 < 500ms |
| 内容可扩展 | 新卡 = 配置数据 + 少量脚本,**不改核心代码**;目标"设计师一天上 10 张卡" |
| 可回放/可调试 | 每局有完整操作日志,可逐回合重放(平衡分析 + Bug 复现 + 反作弊三用) |
| 团队规模假设 | **1~3 人独立团队 + 外包美术**(如果实际不同请告诉我,会显著影响选型) |

**非目标(首发不做)**:跨服全球同服、赛事系统、UGC 卡牌、主机平台。

## 2. 目标平台

### 【待定夺 D1】首发平台

| 选项 | 优点 | 缺点 | 适合场景 |
|------|------|------|----------|
| **A. PC (Steam) 优先,买断或F2P** | 独立卡牌游戏在 Steam 有成熟发现机制(新品节/EA);无需版号即可出海;开发调试成本最低;鼠标操作最适配战棋 | 卡牌对战长线收入天花板低于移动端;国内 Steam 用户面窄 | 小团队验证玩法 |
| B. 移动端优先 (iOS/Android) | 卡牌对战核心付费盘在移动端;碎片时间契合 8~12 分钟局长 | 国内需版号(周期不可控);买量成本高;UI 适配工作量大;上架合规多 | 有发行资源的团队 |
| C. Web 优先 (浏览器) | 零安装、链接即玩、传播快;适合早期约战测试 | 商业化最弱;性能与平台能力受限 | 作为测试渠道而非主平台 |

**我的推荐:A + 为移动端预留设计**。即:首发 Steam(PC),但从第一天起 UI 按"竖屏可改横屏、触控可点选"的规范做(GDD 已按此约束设计操作),美术资源按移动端分辨率规格产出。玩法验证成功后再做移动端移植与版号申请。开发期间可随时出 Web 构建给朋友测试(见 D2,这会影响引擎选择)。

## 3. 游戏引擎选型

### 【待定夺 D2】引擎

回合制 2D 卡牌+棋盘游戏,对引擎的真实需求是:**2D 渲染、UI 系统、动画/特效、多平台导出**。不需要高端 3D、不需要物理。候选:

| 维度 | **Godot 4 (推荐)** | Unity | Cocos Creator |
|------|------------------|-------|---------------|
| 语言 | GDScript / **C#** | C# | TypeScript |
| 2D 与 UI | 2D 是一等公民;Control 节点系统做卡牌 UI 效率高 | 2D 是 3D 的子集;UGUI/UIToolkit 可用但繁琐 | 2D/UI 优秀,为此而生 |
| 费用与许可 | **完全免费开源,无任何抽成** | 有免费额度,商业条款历史上反复变动(2023 信任危机) | 免费,引擎主导权在国内 |
| Web 导出 | 支持(4.3+ 已可用,包体偏大) | 支持但包体大、加载慢 | **最强**(出身即 Web) |
| 移动导出 | 成熟 | **最成熟** | 成熟(国内小游戏渠道独占优势) |
| 生态/资产商店 | 较小但增长快 | **最大** | 中等,中文生态好 |
| AI 辅助开发 | 本工作环境已有 Godot 技能链(GdUnit4 单测 + PlayGodot 自动化 E2E + 一键导出部署),**可显著提高我协助你开发的效率** | 良好 | 良好 |
| 与后端共享规则代码 | 用 **C#** 写规则引擎 → 客户端(Godot C#)与服务端(.NET)共用同一份 DLL | 同左(C# 天然优势) | TS 规则引擎 → Node 服务端共用,同样可行 |
| 主要风险 | C# 移动端导出成熟度略逊 GDScript 路线;招人比 Unity 难 | 许可政策风险;闭源 | 若目标不是国内小游戏渠道,优势用不上 |

**我的推荐:Godot 4 + C#。** 理由:
1. 这个项目对引擎的需求 Godot 全部覆盖,Unity 的生态优势(3D、资产商店)在本项目用不上;
2. **C# 规则引擎可在客户端与服务端之间 100% 共享**(见第 6 节),这是本项目架构的关键决策,Godot C# 与 Unity 在此打平;
3. 零许可成本、开源,对独立团队最安全;
4. 我这边有现成的 Godot 自动化测试与导出工具链,开发迭代(尤其规则引擎的回归测试)会快很多。

**选 Unity 的正当理由**:你或队友已熟练 Unity、或已有 Unity 资产/外包管线。熟悉度 > 引擎优劣,如果是这种情况请直说,方案整体不变,只换客户端载体。

## 4. 联机架构

回合制卡牌对战是联机游戏里架构最友好的品类,这部分**没有太多待定项,直接给方案**:

### 4.1 总体架构:服务端权威 + 命令流

```
┌──────────┐  命令(Command)   ┌───────────────────────┐
│ 客户端 A  │ ───────────────▶ │        对战服务器        │
│ (Godot)  │ ◀─────────────── │  ┌─────────────────┐  │
└──────────┘  事件流(Events)   │  │  规则引擎 (C#)    │  │
┌──────────┐                  │  │  · 校验合法性      │  │
│ 客户端 B  │ ───────────────▶ │  │  · 结算 → 事件序列 │  │
│ (Godot)  │ ◀─────────────── │  └─────────────────┘  │
└──────────┘   WebSocket      │  隐藏信息裁剪 / RNG / 计时 │
                              └───────────────────────┘
```

- **客户端只发送意图**(`打出手牌#3到B2`、`随从@C3移动到C4`),**服务器结算并广播事件序列**(`卡牌打出`、`随从受伤`、`随从死亡`…),客户端按事件播放表现。
- **隐藏信息在服务端裁剪**:对手手牌、牌库顺序永不下发 → 从架构上根除看牌外挂。
- **RNG 全部在服务端**,种子入库,配合命令日志实现**确定性重放**。
- 断线重连 = 下发当前完整快照 + 未播放事件;观战 = 只读事件流(延迟 2 回合)。
- 协议:**WebSocket + 二进制序列化(MessagePack)**。回合制流量极小,无需 UDP/帧同步那一套。

### 4.2 客户端表现层技巧

- **本地预结算**:玩家自己的操作先本地用同一份规则引擎跑一遍立即出表现,服务器确认后对账(不一致则以服务器为准回滚)——消除回合内操作延迟感。
- 对手回合的事件按"表现节奏"排队播放(攻击动画完再播死亡),事件流与动画解耦。

## 5. 后端服务选型

### 【待定夺 D3】对战服务器与平台服务

需要两类东西:①**对战房间服务**(跑规则引擎的实时会话);②**平台服务**(账号、匹配、卡牌收藏、商店、排行榜)。

| 选项 | 组成 | 优点 | 缺点 |
|------|------|------|------|
| **A. 自研 .NET 服务(推荐)** | ASP.NET Core + WebSocket 对战服 + PostgreSQL + Redis | 规则引擎 C# 直接跑在服务端,**一份代码**;回合制并发压力小,单机可扛数千同时对局;完全掌控 | 账号/匹配/商店都要自己写(但都是成熟轮子);需自己运维 |
| B. Nakama (开源游戏后端) | Nakama(Go)+ 其内置账号/匹配/存储/排行 | 平台服务开箱即用,省 1~2 个月;开源可自部署 | 对战逻辑要用 Go 或 Lua/TS 插件写 → **规则引擎无法与 C# 客户端共享**,要么双实现要么客户端弃用预结算;定制深了反而别扭 |
| C. 托管服务 (PlayFab / UOS 等) | 云厂商全家桶 | 运维最省 | 供应商锁定;对战逻辑托管形态受限;成本随规模走高;国内访问质量看厂商 |

**我的推荐:A,自研 .NET。** 决定性理由是**规则引擎单一实现**(第 6 节):它同时是"服务端结算器、客户端预结算器、AI 训练沙盒、平衡模拟器",这个资产的价值远超 Nakama 省下的平台服务开发时间。平台服务(账号/匹配/收藏)对回合制游戏而言就是普通 CRUD + 一个匹配队列,工作量可控。

部署形态:开发期单台云主机 Docker Compose(对战服 + Postgres + Redis);上线后对战服无状态多开 + 网关路由,天花板足够高。

## 6. 核心架构设计:规则引擎

**这是整个项目最重要的技术资产**,与引擎/后端选型解耦,值得单独一节。

### 6.1 原则

- 纯 C# 类库(`HoldTheLine.Rules`),**零 Godot/Unity 依赖、零 IO、零时间依赖**——输入命令,输出事件,完全确定性。
- 同一份 DLL 被四处引用:服务端结算、客户端预结算、AI 对手(直接对规则引擎做搜索)、命令行平衡模拟器(跑十万局出胜率报表)。

### 6.2 核心模型

```csharp
// 状态:不可变快照式(便于回滚、搜索、测试)
GameState { Board(5x4格), Players[2]{ Leader, Hand, Deck, Mana }, TurnInfo, RngState }

// 输入:玩家意图,可枚举、可序列化
Command = PlayCard | MoveUnit | Attack | UseLeaderSkill | EndTurn | Concede

// 输出:结算产生的原子事实,客户端据此播表现
Event = CardPlayed | UnitMoved | UnitDamaged | UnitDied | LeaderDamaged
      | CardDrawn | TurnStarted | GameEnded | ...

// 引擎唯一入口
Resolver.Execute(GameState, Command) → (GameState', Event[])   // 非法命令则拒绝
```

- 所有卡牌效果都通过**事件总线上的触发器**实现(战吼=监听 CardPlayed,亡语=监听 UnitDied),与炉石/杀戮尖塔的成熟范式一致。
- **回放 = 初始状态 + 命令序列重跑**;单元测试 = 构造状态、喂命令、断言事件,GdUnit4/xUnit 均可覆盖。目标:每个关键词、每张卡至少一条结算测试。

## 7. 卡牌数据与效果系统

**目标:上新卡不改引擎代码。** 分两层:

1. **数据层(JSON/表格)**:费用、攻防、关键词列表、稀有度、阵营、美术引用。纯配置,策划可直接编辑。
   ```json
   { "id": "IV_007", "name": "誓约盾卫", "faction": "iron_vow",
     "cost": 3, "atk": 2, "hp": 5, "keywords": ["guard", "hold_fast"],
     "effects": [{ "trigger": "battlecry", "action": "buff",
                   "target": "adjacent_allies", "buff": {"hp": 1} }] }
   ```
2. **效果原语层(C#)**:`damage / buff / summon / move / draw / destroy...` × 目标选择器 `self / adjacent / row / column / half_board / cell...`。约 20 个原语 + 15 个选择器的组合足以表达首发 90% 的卡;剩下 10% 的怪卡允许挂专属 C# 脚本类(白名单注册),不做万能 DSL——**DSL 过度设计是卡牌游戏项目最常见的死法之一**。

配套工具(P1,可后置):卡牌编辑器先用表格 + JSON Schema 校验顶着,不急着做 GUI。

## 8. 客户端架构

- **场景结构**:`Main → 大厅(战线地图/卡组编辑/商店) → 对战场景`。
- **对战场景分层**:
  - `BattleView`(棋盘、棋子、手牌的纯表现节点);
  - `EventPlayer`(把服务器事件序列翻译成动画时间线,队列化播放);
  - `InputController`(拖拽出牌、点选移动,输出 Command);
  - `LocalPredictor`(引用规则引擎 DLL 做本地预结算)。
- **表现关键投入点**(卡牌游戏的手感 80% 来自这里):卡牌拖拽的跟手曲线、合法格高亮、攻击顶撞动画、伤害数字节奏。里程碑 M2 专门留了"手感周"。
- UI 适配:基准 1920×1080 横屏,锚点布局兼容 16:10 与平板比例(为移动端预留)。

## 9. 美术风格

### 【待定夺 D4】美术方向

约束条件:独立团队预算、需要外包、棋盘+棋子+卡面三套资产体系、辨识度。核心问题:**随从在棋盘上如何呈现?** 这是本作和纯卡牌游戏最大的资产差异——每个随从需要"卡面插画"+"场上棋子"两种形态。

| 方案 | 描述 | 优点 | 缺点 | 成本估计 |
|------|------|------|------|----------|
| **A. 桌面战棋质感(推荐)** | 卡面 = 绘画插画;场上棋子 = **立牌式(Standee)**,即插画裁剪 + 底座,像实体桌游的纸立牌;棋盘做实体桌面质感(木纹、战术地图、辉尘刻痕) | 棋子直接复用卡面插画,**资产量减半**;"桌游感"与战棋玩法互文,风格自洽;立牌只需轻量动效(倾斜、顶撞、受击闪白)不需逐帧动画 | 战斗表现上限低于全动画方案;需要用特效/镜头弥补打击感 | 每张卡 1 张插画(外包单价可控),首发 ~160 张 |
| B. 像素风 | 卡面与棋子都是像素;棋子有 4~8 帧动画 | 单价低;独立游戏受众好感;动画成本可控 | 像素卡牌对战已有拥挤感(如 Tavern Rumble);卡面信息密度与美感上限低;"高级感"难做 | 低,但要找到风格稳定的像素画师 |
| C. 全手绘 + 骨骼动画 (Spine) | 炉石级卡面 + 场上 Spine 动画角色 | 表现天花板最高 | 每角色插画+拆件+绑骨+动画,**成本是 A 的 3~5 倍**;独立团队首发 160 张卡不现实 | 高,不推荐首发采用 |
| D. 低多边形 3D 棋子 | 3D 小棋子 + 2D 卡面 | 棋子动画一次绑骨多次复用 | 引入 3D 管线,与 2D 卡面两套审美难统一;外包管理复杂 | 中高 |

**我的推荐:A(桌面战棋质感)**,并保留升级路径——立牌方案未来可对传说卡单独升级 Spine 动画作为付费皮肤(商业化点)。整体氛围基调:**战地手账/作战桌**——泛黄地图、图钉、咖啡渍、辉尘的青金色发光作为唯一魔法色,冷灰暖褐打底。四阵营用主色区分(铁誓=钢蓝、游群=赭红、教团=紫金、匠会=黄铜)。

> 定稿方式建议:选定方向后先出 **1 张概念图(对战场景全景)+ 3 张卡面 + 1 套立牌**的风格探索包,验证后再批量外包。我也可以先用 AI 生成占位美术把整个 demo 铺满,风格探索与玩法验证并行。

## 10. 音频方向

- 音乐:战地管弦 + 低音提琴/军鼓的克制配器,对局尾声(斩杀窗口)动态切激烈层(纵向分层混音,FMOD 或 Godot 原生 AudioBus 均可,回合制用原生够了)。
- 音效优先级:出牌"啪嗒"、棋子移动"叩"、攻击顶撞、辉尘充能——这四个占手感的大头,先做好这四个。
- 首发不做卡牌语音(成本高),给传说卡预留语音位。

## 11. 实现计划与里程碑

以 1~2 人开发 + 我(AI)辅助编码的节奏估计;若人力不同请告知,我重排。

### M0 — 纸面原型(第 1~2 周)
- 实体卡纸 + 打印棋盘,验证 GDD §12 的问题 1/2/3(棋盘尺寸、行动点、守护强度)。
- **产出:锁定核心规则 v1。这是全项目风险最高也最便宜的两周,不可跳过。**

### M1 — 规则引擎 + 本地对战原型(第 3~8 周)
- C# 规则引擎(棋盘/回合/移动/战斗/10 关键词/首发 40 张白板+简单效果卡),xUnit 全覆盖。
- Godot 客户端:同屏热座对战(两人一台机),占位美术。
- 命令行模拟器:随机 AI 互打千局,输出基础数值报表。
- **产出:可玩、可测、可调的单机版。Go/No-Go 决策点:好玩吗?**

### M2 — 联机垂直切片(第 9~16 周)
- .NET 对战服(房间、命令校验、事件广播、断线重连)+ 账号(先只做游客号)。
- 客户端联网 + 本地预结算 + "手感周"(拖拽/高亮/打击感专项打磨)。
- 简单匹配(单队列)+ 好友房间。卡池扩到 ~80 张,4 阵营各有雏形。
- **产出:两台电脑隔着互联网打完整对局。开始小圈子测试(Web 构建发朋友)。**

### M3 — 内容与系统完备(第 17~26 周)
- 卡池 ~160 张(4阵营×30 + 中立40);卡组编辑器;收藏/合成;天梯与赛季框架。
- 规则型 AI(规则引擎上做 1~2 层搜索)用于教学关与人机模式。
