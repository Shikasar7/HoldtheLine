## A1 — 卡组口令(导入/导出)

### A1.1 编解码器(Rules 层)

新文件 `src/HoldTheLine.Rules/Cards/DeckCode.cs`(Rules 层无新依赖,压缩用 BCL
`System.IO.Compression.DeflateStream`):

```csharp
public enum DeckCodeError { None, BadFormat, UnsupportedVersion, DataMismatch }

public sealed record DeckCodePayload(
    string Rules,                    // 生成时的 RulesInfo.Version
    string Hash,                     // 生成时 DataHash 的前 8 位 hex
    string Leader,
    IReadOnlyList<string> Cards);

public static class DeckCode
{
    public const string Prefix = "HTL1-";

    /// dataHash 传完整 64 位 hex,内部只取前 8 位入码。
    public static string Encode(string leader, IReadOnlyList<string> cards,
                                string rulesVersion, string dataHash);

    /// 只做结构解码:前缀/Base64Url/Deflate/json 任一失败 → BadFormat;v != 1 → UnsupportedVersion。
    public static (DeckCodeError Error, DeckCodePayload? Payload) Decode(string code);

    /// 环境校验:payload.Rules != rulesVersion 或 payload.Hash != dataHash[..8] → DataMismatch。
    public static DeckCodeError Check(DeckCodePayload p, string rulesVersion, string dataHash);
}
```

- **载荷 json**(内部私有 DTO,`JsonPropertyName` 固定,紧凑序列化不缩进):
  `{"v":1,"rules":"0.4.1","hash":"a1b2c3d4","leader":"...","cards":["..",".."]}`
- **管线**:UTF-8 bytes → `DeflateStream`(Compress)→ Base64 → URL 安全化
  (`+`→`-`,`/`→`_`,去掉尾部 `=`)→ 前缀 `HTL1-`。Decode 逆向(补 `=` 填充)。
- 口令内**不含卡组名**(有意为之,导入方自己命名);不用 `RulesJson.Options`
  (避免与规则契约耦合,DTO 自带属性名)。
- 卡组合法性(30张/稀有度/阵营)**不在 DeckCode 里查**,由调用方跑 `DeckValidator.Validate`。

测试:新文件 `src/HoldTheLine.Rules.Tests/DeckCodeTests.cs`
1. roundtrip:Encode→Decode 得到相同 leader/cards/rules/hash;
2. 篡改中段字符 → `BadFormat`(Deflate/json 必炸,捕获为错误而非异常外泄);
3. 无前缀 / 空串 / 纯垃圾 → `BadFormat`;
4. 手工构造 `"v":2` 载荷 → `UnsupportedVersion`;
5. `Check`:rules 不同 → `DataMismatch`;hash 前 8 位不同 → `DataMismatch`;全同 → `None`。

## B1 — 注册 / 登录(协议 v5)

### B1.1 协议(Messages.cs 冻结面变更,本节即批准范围)

`ProtocolJson.cs`:`ProtocolVersion = 4` → `5`。

`Messages.cs` 新增(client 侧挂 `[JsonDerivedType(typeof(Register), "register")]`、
`[JsonDerivedType(typeof(Login), "login")]`,server 侧挂 `[JsonDerivedType(typeof(AuthOk), "auth_ok")]`):

```csharp
/// 把当前已握手的持久身份升级为用户名+密码账号(战绩/卡组/评分原地保留)。
public sealed record Register : ClientMessage
{
    public required string Username { get; init; }
    public required string Password { get; init; }
}

/// 用已注册账号登录:成功后本连接身份切换为该账号,并轮换 secret(见 B1.3)。
public sealed record Login : ClientMessage
{
    public required string Username { get; init; }
    public required string Password { get; init; }
}

public sealed record AuthOk : ServerMessage
{
    public required string Username { get; init; }
    /// 仅 login 返回:该账号的 guest_id 与新轮换的 secret,客户端持久化进 identity.json。
    public string? GuestId { get; init; }
    public string? Secret { get; init; }
}
```

失败一律复用现有 `ErrorMsg`(带 Seq),错误码固定:
`name_taken` / `weak_password` / `bad_credentials`(不区分「无此用户」与「密码错」,防枚举)/
`too_many_attempts` / `not_identified`(匿名连接不能注册)/ `already_bound`(该身份已绑定过用户名)。

**secret 轮换语义(设计决定,写进代码注释)**:登录成功即生成新 secret 替换库中哈希——
旧设备的 identity.json 随之失效,下次 hello 收 `bad_identity` 后需重新登录。单活跃设备
凭据,轮换即踢旧,安全且实现最简。

### B1.2 服务器

**`Data/AccountStore.cs` 扩展**:
- 建表后跑迁移:`ALTER TABLE accounts ADD COLUMN username TEXT` / `username_lower TEXT` /

```csharp
public enum AuthOutcome { Ok, NameTaken, BadCredentials, NotIdentified, AlreadyBound }

/// register:给 guestId 绑定用户名+密码。username_lower 撞唯一索引 → NameTaken;
/// 该行已有 username → AlreadyBound。
public AuthOutcome BindCredentials(string guestId, string username, string password);

/// login:按 username_lower 查行,PasswordHash.Verify 通过后生成新 secret(32 hex),
/// secret_hash 更新为其 SHA256,返回 (Ok, guestId, newSecret);查无/验证失败 → BadCredentials。
public (AuthOutcome Outcome, string? GuestId, string? NewSecret) Login(string username, string password);
```

**新文件 `Server/PasswordHash.cs`**:
```csharp
public static class PasswordHash
{
    // 格式 "pbkdf2$210000$<salt b64>$<hash b64>";PBKDF2-SHA256, 盐16B, 输出32B,
    // 迭代 210_000(OWASP 2023 建议线)。Rfc2898DeriveBytes.Pbkdf2 静态 API。

    public static bool Verify(string password, string stored); // CryptographicOperations.FixedTimeEquals
}
```

**新文件 `Server/AuthThrottle.cs`**(DI 单例):

        SearchTopK = 2, SearchRollout = 4 };
    public static readonly AiProfile Hard   = new() { Level = AiLevel.Hard,   UseSearch = true,
        SearchTopK = 4, SearchRollout = 8 };   // == SearchAi 现默认,行为不变

### C4 — 难度校准(实现完 C2+A2 后跑,结果回写本文档)

用 Sim 三场对照(同一对内置预组、≥200 局、`--parallel 8`、双种子各跑一次):
1. `--ai0 search --ai1 fast` → 期望 search ≥65%;
2. `--ai0 fast --ai1 easy` → 期望 fast ≥65%;
3. `--ai0 search --ai1 easy` → 期望 ≥80%(sanity)。

不达标时的**允许调参范围**:`AiProfile.Easy.Epsilon`(0.15~0.35)、
`Normal.SearchTopK`(1~3)、`Normal.SearchRollout`(2~6)。调完把最终数值与三场胜率
回写到本节表格并 commit。**不许**为拉开差距去改 SearchAi 的评估函数或 GreedyAi。

**实测(2026-07-19,iron_wall vs wildpack_hunt、200 局/种子、`--parallel 8`、种子 1001/2002)**:
三档一次达标,**无需调参**(AiProfile 默认值保留)。数值含先手优势(强策略固定坐 seat0),
但档差足够大、双种子一致。

| 对照 | 目标 | 实测(seat0 胜率,种子 1001 / 2002) |
|---|---|---|
| Hard vs Normal (search vs fast) | ≥65% | **71.0% / 69.5%** ✅ |
| Normal vs Easy (fast vs easy) | ≥65% | **81.5% / 77.5%** ✅ |
| Hard vs Easy (search vs easy) | ≥80% | **94.5% / 94.0%** ✅ |
