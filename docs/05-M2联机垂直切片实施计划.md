# 《守线 / Hold the Line》M2 联机垂直切片实施计划

> 状态:Go 决策已于 2026-07-17 做出(原型试玩通过 DoD)。本文档由 Fable 5 产出,
> 交由 Opus 4.8 按 §11 阶段划分实施。上游文档:技术 PRD §4/§5/§11(M2)、原型计划 §3.1。

---

## 1. M2 的定义:做什么,不做什么

**一句话目标:两台电脑隔着互联网,打完一整局《守线》,断线能重连。**

### 1.1 In(必须交付)

- 自研 .NET 对战服务器(PRD D3 选项 A):房间、命令校验、事件广播、断线重连。
- Godot 客户端联网:`RemoteGameHost` 实现 `IGameHost`,BattleScene 基本不动。
- 好友房间(房间码创建/加入)+ 游客账号(无密码)。
- 席位 1 视角渲染(联机后本地玩家可能坐在任一席位)。
- 手感周:拖拽跟手、攻击顶撞动画、伤害数字节奏专项打磨。
- 回合计时器(服务端权威,防 AFK)。

### 1.2 Out(明确不做,防蔓延)

- 匹配队列(单队列排位)——好友房间跑通后若有余力再加,不算 M2 验收项。
- 本地预结算 + 对账回滚(PRD §4.2)——回合制 + 命令流,实测延迟感可接受再说;
  架构上 `IGameHost` 已为它留好了缝,M3 再做不欠债。
- MessagePack 二进制协议——M2 用 JSON(`RulesJson` 契约现成、可读、易调试);
  换 MessagePack 是纯优化,流量对回合制毫无压力。
- 账号密码 / 第三方登录、卡牌收藏、商店、排行榜(M3)。
- 观战、回放 UI(数据层已支持——Config + 命令日志,UI 是 M3 的事)。
- **Web 构建**:PRD M2 里"Web 构建发朋友"一条**作废**——Godot 4.x 的 C# 尚不支持
  Web 导出。分发继续走 Windows zip(原型期管线已跑通)。

### 1.3 M2 要回答的问题

1. 隔互联网打一局,操作反馈延迟是否可接受?(决定 M3 是否要做预结算)
2. 断线重连的体验是否达到"掉线不毁局"?
3. 打磨后的手感是否达到"愿意让朋友反复开局"?

## 2. 起跑线:原型期五条硬约束的兑现情况

这是 M2 成本远低于常规估计的原因,实施前先确认这些资产还在(跑一遍全部测试):

| 硬约束(原型计划 §3.1) | 现状 | M2 直接收益 |
|---|---|---|
| ① 可序列化 Command/Event + 往返测试 | ✅ `SerializationTests` | 协议消息体零新增设计,直接上线缆 |
| ② 回环序列化(每条命令/事件过 JSON 边界) | ✅ `LocalGameHost.LoopbackSerialization` | 客户端不可能依赖共享引用,联网无惊喜 |
| ③ 按席位裁剪(`GameEvent.RedactFor`) | ✅ | 服务端裁剪即调用现成方法,看牌外挂架构性根除 |
| ④ 确定性 RNG + 命令日志(`MatchConfig` + `CommandLog`) | ✅ | 服务端存档 = 完整重放;bug 报告可复现 |
| ⑤ 异步 `IGameHost` | ✅ | BattleScene 已按"结果走事件流"写,换宿主即可 |
| 表现层与规则层零耦合 | ✅ `ArchitectureTests` 扫描强制 | BattleScene 不认识 GameState,天然是"瘦客户端" |

**结论:M2 的核心工作不是"给游戏加联机",而是"写一个把 `LocalGameHost` 搬到
进程外的传输层"。** 规则引擎一行不改。

## 3. 总体架构与工程布局

```
┌────────────────────┐   WsEnvelope(JSON)   ┌──────────────────────────────┐
│ Godot 客户端 A       │ ◀──────────────────▶ │ HoldTheLine.Server (ASP.NET) │
│  BattleScene        │      WebSocket       │  ┌────────────────────────┐  │
│   └ RemoteGameHost ─┤                      │  │ Room / MatchSession    │  │
├────────────────────┤                      │  │  └ LocalGameHost ◀── 原样复用│
│ Godot 客户端 B / Bot │ ◀──────────────────▶ │  │     (规则引擎+裁剪+日志) │  │
└────────────────────┘                      │  └────────────────────────┘  │
                                            └──────────────────────────────┘
```

关键决定:**服务端的对局宿主就是 `LocalGameHost` 本身**。它已经是"权威状态 + 席位
裁剪 + 命令/事件日志"的完整实现,`MatchSession` 只做三件事:把 WebSocket 消息翻译成
`SubmitCommandAsync`、把 `Subscribe` 的事件推回连接、管理座位与重连令牌。

### 3.1 新增工程

```
src/
├── HoldTheLine.Net/            # 共享协议层(客户端与服务器同用,禁止引用 Godot)
│   ├── Protocol/               #   WsEnvelope、全部 C→S / S→C 消息 record
│   ├── ProtocolJson.cs         #   复用 RulesJson.Options + 协议消息的多态注册
│   └── Client/
│       ├── GameServerClient.cs #   ClientWebSocket 封装:连接/心跳/收发循环/重连
│       └── RemoteGameHost.cs   #   IGameHost 实现(见 §6)
├── HoldTheLine.Server/         # ASP.NET Core minimal API + WebSocket
│   ├── Program.cs              #   /ws 端点、配置、优雅停机
│   ├── Rooms/RoomManager.cs    #   房间码 → Room;创建/加入/清理
│   ├── Rooms/MatchSession.cs   #   一局:2 席位连接 ↔ LocalGameHost;计时器;重连
│   └── SessionAuth.cs          #   游客 id + 席位令牌(resume token)
├── HoldTheLine.BotClient/      # 控制台机器人:GreedyAi 走 RemoteGameHost 联网打
└── HoldTheLine.Server.Tests/   # 进程内集成测试(见 §8)
```

依赖方向(`ArchitectureTests` 加断言):
`Rules ← Net ← {Server, BotClient, game}`;`Server` 不得引用 Godot;
`game/scripts` 允许新增引用 `HoldTheLine.Net.Client.*` 与 `Protocol` 消息,
**仍然禁止** GameState/Resolver 等(现有扫描规则不放松)。

全部 net8.0,与现有工程一致(Godot 4.6 .NET 支持 net8)。

## 4. 协议设计 v0

### 4.1 信封与序列化

```jsonc
{ "t": "submit_command", "seq": 12, "p": { /* 消息体,snake_case */ } }
```

- 一条 WebSocket Text 帧 = 一个 `WsEnvelope { string T; int Seq; JsonElement P }`。
- 序列化全部走 `RulesJson.Options`(命令/事件的多态转换器已注册,协议消息在
  `ProtocolJson` 补注册)。
- `seq` 客户端自增,服务器应答带回 `ack_seq`,便于对请求-响应配对与日志排查。
- 版本握手:`hello` 携带 `protocol_version`(常量,起始 1)与 `rules_version`
  (`RulesInfo.Version`);任一不匹配 → `error{code:"version_mismatch"}` 后关闭。
  这是防"客户端旧规则 vs 服务端新规则"静默劈叉的唯一闸门,**必须做**。

### 4.2 消息目录

**C→S**

| t | 载荷 | 说明 |
|---|---|---|
| `hello` | guest_id, name, protocol_version, rules_version, resume_token? | 连上必发第一条;带 resume_token 则走重连分支 |
| `create_room` | deck_id | 返回房间码,等待对手 |
| `join_room` | code, deck_id | 双方就位即开局 |
| `leave_room` | — | 开局前退出等待 |
| `submit_command` | command(多态 Command) | 唯一的对局输入;含 Concede/EndTurn |
| `resync` | since_event_index | 主动请求快照+补事件(重连或自检不一致时) |
| `ping` | — | 心跳,20s 间隔;服务器 40s 无心跳判掉线 |

**S→C**

| t | 载荷 | 说明 |
|---|---|---|
| `hello_ok` | server_time, motd? | |
| `room_created` | code | 展示给房主转发朋友 |
| `match_started` | seat, resume_token, view(PlayerView), opponent_name | 开局快照 |
| `command_result` | ack_seq, accepted, error? | 对应 `SubmitCommandAsync` 的返回值 |
| `events` | batch(GameEvent[]), event_index, legal_commands? | 结算事件批;若接收方成为行动席位,附带其合法命令(见 §6.2) |
| `resync_ok` | view, events_since, event_index, legal_commands? | 重连/自检应答 |
| `opponent_status` | connected(bool), grace_seconds? | 对手掉线/回来了 |
| `turn_timer` | seat, seconds_left | 计时提示(每 10s 或剩 10s 内每秒) |
| `match_ended` | winner_seat, reason(normal/concede/timeout/abandon) | 终局(常规胜负也会先有 MatchEndedEvent,这条是会话层收尾) |
| `error` | code, message | version_mismatch / room_not_found / room_full / not_your_turn 等 |

### 4.3 一局的时序(黄金路径)

```
A: hello → hello_ok → create_room → room_created(code=X7K2QF)
B: hello → hello_ok → join_room(X7K2QF)
S: → A: match_started(seat=0, view…, legal_commands)   → B: match_started(seat=1, view…)
A: submit_command(PlayCard…) → S 校验结算
S: → A: command_result(accepted) + events(裁剪@0)      → B: events(裁剪@1)
…… 直至 MatchEndedEvent → match_ended → 双方可留在房间"再来一局"(复用房间,重开 MatchSession)
```

## 5. 服务端设计

### 5.1 组成

- **Program.cs**:minimal API;`GET /healthz`;`GET /ws` 升级 WebSocket;
  配置(端口、回合时限、掉线宽限)走 appsettings + 环境变量。
  卡牌/领袖数据从 `game/data` 同一份 JSON 加载(路径可配)——**单一数据源,禁止复制卡表**。
- **RoomManager**:`ConcurrentDictionary<string, Room>`;房间码 6 位大写字母数字
  (去易混淆字符 0/O/1/I);空房 10 分钟回收;对局结束后房间保留供"再来一局"。
- **MatchSession**:每局一个;内部就是 `new LocalGameHost(db, leaders, config)`
  (Seed 用服务端 CSPRNG 生成)。持有两个席位的连接槽;
  `Subscribe(seat, …)` 两路事件各自序列化推送;命令入口做**三层校验**:
  连接→席位映射正确 → `command.Seat == 席位`(host 里已有第二道)→ 交给 Resolver。
- **并发模型**:每个 MatchSession 一个 `Channel<(conn, envelope)>` 单消费者循环,
  天然串行化一局内的所有输入,不与 `LocalGameHost` 的锁较劲。房间之间完全并行。

### 5.2 计时与掉线语义(默认值,可配)

| 事项 | 规则 |
|---|---|
| 回合时限 | 90s;超时服务器代提交 `EndTurnCommand`,并发 `turn_timer` 预警 |
| 连续超时 | 同一玩家连续 2 次超时 → 代提交 `ConcedeCommand`(reason=timeout) |
| 掉线 | 立即通知对手 `opponent_status(connected=false, grace=120)`;对局暂停计时 |
| 宽限内重连 | resume_token 验证 → `resync_ok`(快照+补事件)→ 对局继续 |
| 宽限耗尽 | 判掉线方 Concede(reason=abandon) |
| 服务器重启 | M2 不做对局持久化恢复;进行中的局判平结束并致歉。命令日志落盘(JSONL)供重放排查 |

### 5.3 安全底线(M2 级别)

- 服务端权威 + 席位裁剪已根除看牌/改数值类外挂;
- 游客 id 无认证,resume_token 是唯一会话凭据(128bit 随机,只在 match_started 下发);
- 不做限流/反 DDoS(朋友圈测试规模),但房间数、每连接消息速率设个宽松上限防意外死循环。

## 6. 客户端设计

### 6.1 `IGameHost` 的最小扩展

现状:BattleScene 用了 `LocalGameHost` 的两个接口外方法 —— `LegalCommands(seat)`
与 `SuggestCommand(seat)`。处理:

- `LegalCommands(int seat)` **提升进 `IGameHost`**(联机时读 RemoteGameHost 缓存,见下);
- `SuggestCommand` 不进接口(AI 建议是本地宿主特有);BattleScene 改为
  `private IGameHost _host; private LocalGameHost? _localHost;`,vs-AI/热座时二者同源。
- 接口文档补一条契约:**`Subscribe` 回调可能来自任意线程**(远程宿主从 WS 接收线程
  派发)。BattleScene 的 `_pendingEvents` 相应改为 `ConcurrentQueue<GameEvent>`,
  `AccumulateStat` 挪到主线程消费侧(现在它在回调里改 int[],联机下就是数据竞争)。

### 6.2 `RemoteGameHost`:同步读缓存 + 异步写

`IGameHost` 的读方法(`GetView`/`GetEventLog`/`LegalCommands`)是同步的,而数据在
服务器。解法不是把接口改异步(会重写 BattleScene),而是**服务器推、客户端缓存**:

- `match_started`/`events`/`resync_ok` 都携带(或伴随)最新裁剪视图与
  (轮到你时的)合法命令 → RemoteGameHost 原子更新 `_cachedView`/`_cachedLegal`/
  `_eventLog`,读方法只读缓存;
- `SubmitCommandAsync` 发 `submit_command`,用 `seq→TaskCompletionSource` 字典等
  `command_result` 兑现返回值(带 10s 超时);
- 事件按 `event_index` 严格递增消费,发现空洞 → 自动 `resync`(自愈);
- 心跳、断线检测、指数退避自动重连(带 resume_token)都在 `GameServerClient`,
  对 BattleScene 只暴露 `ConnectionState` 变化事件(UI 显示"重连中…"横幅)。

> 带宽注:每次事件批都附带全量 PlayerView 是偷懒但正确的做法,单局流量也不过几百 KB,
> M2 不做增量视图。合法命令只发给行动席位,非行动方 `LegalCommands` 返回空(与
> LocalGameHost 行为一致)。

### 6.3 UI 流程改动

- **主菜单**新增「联机对战」→ 子面板:昵称(存 user://)、服务器地址(默认官方地址,
  可编辑,便于 LAN/自建)、〔创建房间〕→ 显示房间码等待 /〔加入房间〕→ 输房间码;
  双方各自在此面板选卡组。
- **BattleScene**:
  - 联机模式 `ViewSeat = 本地席位`(match_started 下发)。**席位 1 视角必须镜像渲染**
    ——己方永远在下。vs-AI 人类固定 seat0,这条路径原型没走过,是客户端最大的新工作量;
    验收:seat1 玩家看到自己的底行在屏幕下方,拖牌/移动/攻击全部正确。
  - 对手回合禁互动(现有 vs-AI 已有此路径,复用);
  - 新 UI:回合倒计时、对手掉线横幅、"重连中…"遮罩、终局后〔再来一局〕(双方都点则
    同房间重开)。
- **GameConfig** 加 `SetOnline(serverUrl, nickname, roomAction…)` 分支。

## 8. 测试策略

原型 150+ 条规则测试全部继续有效。新增三层:

1. **协议单测**(HoldTheLine.Server.Tests):每种消息 JSON 往返;版本不匹配握手拒绝;
   信封对未知 `t` 的容错(忽略+log,不断连)。
2. **进程内集成测试(重头)**:用 `WebApplicationFactory`/TestServer 起完整服务端,
   两个 `RemoteGameHost` + `GreedyAi` 连上打**整局**,断言:
   - 双方事件流各自裁剪正确(seat0 收到的对手抽牌事件不含卡牌 id);
   - 终局一致(双方 MatchEndedEvent 相同胜者);
   - 服务端命令日志重放(`MatchConfig`+CommandLog 喂 LocalGameHost)得到相同终局
     ——**确定性在网络路径上的端到端验证**;
   - 中途硬断一个客户端 → resume_token 重连 → 补齐事件后照常打完;
   - 回合超时 → 自动 EndTurn;连续两次 → 判负。
   这组测试是 M2 的护城河,优先于一切手工验证。
3. **手工冒烟**:
   - 本机:服务器控制台 + Godot 窗口 + **BotClient**(联网版陪练,顺手交付人机联调工具);
   - 双窗口:同机开两个 Godot 实例互打(注意两实例共享 user:// 与 MCP 文件通道——
     MCP 自动化只驱动其一,另一个手动或用 BotClient 代替);
   - LAN:两台机器;
   - 互联网:服务器部署到公网(§9),两地各一人打完整局 → 即 M2 DoD 现场。

## 10. 手感周(N4)工作单

> 2026-07-18 修订:并入首次真人联机实测(公网 VPS)后用户的三条反馈;
> 原 6 项细化为下表,※ 标注的是实测反馈直接点名的项。

目标不是"更多功能"而是"同样的操作更舒服"。逐项验收,全部以录屏对比:

1. **拖拽跟手曲线**:卡牌跟随用临界阻尼插值(而非硬贴鼠标),离手回弹;拖起时卡牌
   放大 1.06 + 轻微倾斜;合法格在拖起瞬间亮,悬停格高亮加强。
2. ※ **攻击动作三段化**:蓄力前摇(向后微缩 ~0.1s)→ 冲撞急进急回(40% 距离,~0.18s)
   → 命中帧屏幕微震(2~3px, 0.1s)。**远程攻击改为投掷物**:从攻击方飞出一枚弹体
   (占位:发光小方块/箭形,~0.25s 直线飞行),命中瞬间才结算受击表现——不再是无实体的瞬间掉血。
3. ※ **受击反馈强化**:受击方闪白 + 后顿(向受力方向位移 6~8px 回弹)+ 短促受击音;
   持盾吸收时改为蓝闪 + 「盾」字浮标,与真实掉血明确区分。
4. **伤害数字节奏**:命中帧弹出,上飘+重力回落,大伤害字号加大;多段伤害错开 80ms。
5. ※ **对手出牌展示**:对手打出任何牌时,屏幕中央短暂展示该卡牌面(放大卡图+名称+效果文本,
   ~1.2s 或点击跳过),再播放落地/效果——否则联机对手根本不知道你打了什么。
   指令牌(无实体落地)尤其依赖这一条。实现:CardPlayedEvent 驱动,复用手牌卡面渲染。
6. **死亡与破线**:死亡淡出+碎裂感(占位实现:缩放+旋转+透明);破线打脸给全屏边缘
   红光脉冲 + 领袖头像抖动(这是核心奖励时刻,值得最重的反馈)。
7. ※ **音效全套**(SfxBank 从 4 条扩到 ~12 条,仍程序合成或免费素材):
   近战命中 / 远程发射+命中 / 指令施放 / 单位死亡 / 领袖受创(已有,加重) /
   回合开始 / 压力潮汐 / **胜利** / **失败** / 抽牌 / 部署落地(已有) / 按钮点击。
   胜负各配一段 2~3 秒的短乐句(合成琶音即可),结算界面同步出现。
8. **回合切换**:横幅动画 + 回合音;压力潮汐结算给专属音/视觉提示(接第 7 条素材)。
9. **事件播放队列**:对手回合事件按表现节奏排队(攻击动画完再播死亡),队列与网络
   到达解耦——第 2/3/5 项都建立在它之上,优先级最高、先做。
